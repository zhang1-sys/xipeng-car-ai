# 小鹏公开 Demo 当前任务板

## 1. 当前判断

本文档从现在开始继续作为仓库里唯一有效的任务板，但目标判断已经更新。

当前项目不是“只差部署包装”，而是进入了**最后一轮产品重构与 bug 收口**：

1. 公网 URL 已有，但当前体验还不够稳，不适合把“能访问”当成“可交付”。
2. 前端首屏和主工作区信息密度过高，页面结构失焦。
3. 智能问答在推荐场景下仍然会漂移，不能稳定承担面试开场主链路。
4. 配置器虽然接口可用，但交互形态仍然像“聊天驱动的状态机”，不够像真正可演示的产品。
5. 仍存在一批结构性 bug 和工程一致性问题。

当前目标仍然是：

**做出一个面试官可直接访问、且主链路稳定可演示的公开 demo。**

但达到这个目标前，必须先完成一轮收敛和重构，而不是继续叠加功能。

---

## 2. 当前最主要的问题

### 2.1 前端结构过乱

当前首页把以下内容同时堆在一个页面里：

1. 顶部导航 CTA
2. Hero 说明区
3. 场景说明卡片
4. 右侧配置摘要卡
5. 聊天主区域
6. 聊天空态推荐入口
7. 侧栏“使用建议 / 你会得到什么 / 常用动作”
8. 多处重复 CTA

结果不是“信息完整”，而是用户第一次进入时难以判断应该先点哪里。

### 2.2 问答入口不稳定

当前推荐、对比、服务问答共用 `/api/chat`，而首页按钮并没有稳定地把用户带入明确模式。

结果是：

1. 推荐问题偶发偏到知识型回答。
2. 面试时如果拿自由问答做开场，命中率不够稳。
3. “推荐 -> 对比 -> 配置器”没有被 UI 明确地设计成受控流程。

### 2.3 配置器形态不对

当前配置器的后端状态机能跑，但前端体验仍偏向“继续聊一句”。

这会导致：

1. 用户不清楚当前处于哪一步。
2. 选项不是“点选为主”，而是要靠文本输入补齐。
3. 配置摘要和聊天消息重复展示，增加视觉噪音。
4. 配置器不像电商/汽车行业里真正可上线的配置器。

### 2.4 存在结构性 bug

当前至少有这几类问题：

1. 同一个 `sessionId` 同时承载普通问答和配置器，会造成上下文串味。
2. 配置器结果同时出现在右侧摘要区和聊天区，造成重复渲染。
3. `backend eval` 仍有 2 个 critical case failed。
4. 运行日志中仍出现 LLM brand inference timeout，说明部分链路稳定性不足。
5. 文档状态已经落后于真实状态，容易误导开发决策。

---

## 3. 结论：离目标还有多远

如果按“能打开公网网页”衡量，已经很近。

如果按“面试官会觉得这是一个产品，而不是一个堆功能的原型”衡量，还差一轮关键重构。

当前更准确的判断是：

**离“可访问 demo”很近，离“足够拿得出手的面试 demo”还有一轮高优先级整治。**

这轮整治不需要扩很多新能力，但必须把以下 4 件事做扎实：

1. 前端减法
2. 问答收敛
3. 配置器重做
4. bug 清扫

## 5. Claude Code 执行清单

Claude Code 后续不再做“扩功能”，只按下面顺序写代码。

### 任务 1：首页减法重构

目标：

把首页从“信息堆叠页”改成“单一主任务入口页”。

要求：

1. 首屏只保留一个主标题、一段边界说明、两个主按钮：
   - 开始选车推荐
   - 直接进入配置器
2. 删除或下沉重复信息块：
   - “使用建议”
   - “你会得到什么”
   - 过多的空态提示卡
   - 多处重复 CTA
3. 保留公开快照 / 未接内部系统说明，但压缩为一块清晰 notice。
4. 页面默认视觉重点只能有一个，不允许 Hero、摘要卡、聊天区同时争抢注意力。

验收：

1. 用户进入页面 3 秒内知道第一步该做什么。
2. 首页首屏无需滚动即可理解产品边界和主动作。

### 任务 2：把问答入口改成受控模式

目标：

不要让自由问答决定演示成败。

要求：

1. 前端新增明确模式切换：
   - 推荐
   - 对比
   - 服务
2. 首页主按钮、快捷入口、跟进按钮都要显式带 mode。
3. `postChat()` 支持传入 `mode`，后端优先使用 forced mode。
4. 推荐入口默认只走推荐模式，不再把“推荐”交给 ReAct 自由发挥。
5. 对比入口默认只走对比模式。
6. 服务问答单独保留，不干扰购车主链路。

验收：

1. “预算 20 万，广州通勤”稳定进入 recommendation。
2. “对比 G6 和 G9”稳定进入 comparison。
3. 快捷入口的结果模式与入口意图一致。

### 任务 3：重做配置器前端

目标：

把配置器从“聊天状态机”改成“可点击的多步配置器”。

要求：

1. 配置器采用明确步骤：
   - 车型
   - 版本
   - 外观
   - 内饰
   - 套件
   - 配置摘要
2. 每一步优先使用按钮 / 单选卡片，不依赖用户手打文本。
3. 当前步骤、已完成步骤、下一步要非常清楚。
4. 配置器结果不要再在聊天区重复展示。
5. 聊天区如果保留，只允许作为“解释当前选择”的辅助区域，不再承担主交互。
6. 配置器完成后只保留一个摘要区，再接试驾 / 门店 / 权益 CTA。

验收：

1. 不输入文字也能走完整个配置流程。
2. 一个没看过项目的人能在 1 分钟内完成一次配置。
3. 配置摘要只出现一次。

### 任务 4：清理主链路 bug

目标：

把“能跑”变成“稳定演示”。

要求：

1. 普通问答和配置器拆分独立会话状态，不能共用一个 session 池里的同一段流程上下文。
2. 修掉配置器重复渲染和状态串味。
3. 修掉 eval 的两个 CRM case：
   - `/api/crm/ack`
   - `/api/crm/callback`
4. 处理明显超时 / fallback 行为，至少让主链路不受 brand inference timeout 干扰。
5. 修完后重新跑：
   - `frontend lint`
   - `frontend build`
   - `backend smoke`
   - `backend eval`

验收：

1. `backend eval` 恢复为 releaseReady=true，或至少明确对齐新的预期并消除 critical failed。
2. 主链路手工跑一遍无明显错乱。

---

## 6. 我来把控的标准

后续我只按下面标准判断是否继续：

1. 页面是否更清楚，而不是更花。
2. 用户是否更少思考“下一步点哪里”。
3. 推荐是否更稳，而不是更会说。
4. 配置器是否更像产品，而不是更像聊天。
5. bug 是否真的减少，而不是被文案掩盖。

只要这 5 条没明显改善，就不算完成。
